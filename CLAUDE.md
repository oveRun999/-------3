# Claude Code 使用量最適化ガイドライン

## 📋 プロジェクト概要
- フロントエンド（Nuxt3）の開発・学習支援
- 使用量上限に達しないよう、トークンを効率的に消費する

## 🧱 技術スタック
- フレームワーク: Nuxt3（Vue3 Composition API）
- 環境: Windows 11
- スキルレベル: フロントエンド初心者（バックエンド・インフラ知識なし）

## 🚫 アクセス禁止ディレクトリ
- `node_modules/`, `.nuxt/`, `.output/`, `dist/`, `.git/`

---

## ⏱ 使用量の仕組み（必読）
- 制限は「深夜リセット」ではなく **最初のリクエストから5時間のローリングウィンドウ**
- Claude.ai（Web/デスクトップ）と Claude Code の使用量は**共有**される
- `ANTHROPIC_API_KEY` が設定されていると**サブスク外の従量課金**になるため注意

---

## 💰 トークン節約ルール（Claude は必ず守ること）

### コンテキスト管理
- 回答に必要なファイルだけ読む。関係のないファイルを広範スキャンしない
- `ls -R` や全体 `grep` を指示なしに繰り返さない
- ファイル全体を読む前に `grep` で必要行だけ確認する

### 検索効率
- `grep`/`find` はディレクトリを絞り込んで実行
- `node_modules` などビルドディレクトリを除外すること

### 変更粒度
- 大規模リファクタリングは小さなステップに分割し、都度確認を取る
- 変更がないファイルを再読み込みしない
- 外部API呼び出しのループはユーザー確認なしに続けない

### 回答スタイル
- 冗長な前置き・まとめは省く。変更点と結論だけ簡潔に述べる
- コード内コメントは最小限（自明な処理には不要）

## 🔧 セッション運用ルール

| 場面 | コマンド |
|------|----------|
| 別タスクに切り替えるとき | `/clear`（完全リセット） |
| 同じタスクが長くなったとき | `/compact 実装済みファイルと次のタスクを必ず保持すること` |
| トークン残量の確認 | `/context` |

- コンテキストが **70% に達したら `/compact`** を使う（83.5%＝約167,000トークンで自動コンパクション）
- 1セッションの目標は **30,000トークン以内**
- タスク完了・切り替え時は必ず `/clear`

---

## 🎯 指示の出し方（Claudeへのヒント）

```
# 悪い例（余計な探索が発生する）
「このコードをよくして」

# 良い例（スコープが明確）
「@src/pages/index.vue のレイアウト崩れを修正してください。変更はこのファイルだけです」
```

- `@ファイルパス` で直接指定するとトークン節約になる
- 無関係なタスクを1セッションにまとめない

| モデル | 使いどき |
|--------|----------|
